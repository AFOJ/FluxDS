## The Design Token Pipeline

 I started this challenge by working on the design token to theme workflow. My first step was to analyse the JSON and understand how the linking worked. The documentation mentioned that not everything in the JSON was required, so I focused on identifying the parts which were most relevant to the Figma designs. I initially considered using Style Dictionary, as it is an industry standard, but after a short attempt with little progress I decided a custom script would be more effective (Time-wise) for this task.

I chose to create a script that converts the themes into CSS variables. I was comfortable with this approach as I have experience traversing complex documents, such as working with Zod safeParse errors and handling error objects while working with the Xero API. The structure felt similar. I identified the main entry point as the `Mapped/{Brand}` object. My first approach was to pass an array of brand targets, such as `BrandA` and `BrandB`, that matched the keys under `Mapped/{Brand}`. The goal was to loop through the `Mapped/{Brand}` until a value was found. This worked, but the values were reference strings like `“Colours.red”`. I needed a way to resolve these.

To resolve these references, I first considered full recursion across the entire document to find the first matching definition. After brainstorming with AI, I refined this approach by limiting traversal to specific paths, as references only existed within certain sections of the JSON. This proved successful and allowed me to fully resolve the mapped values. Some resolved values were not in an ideal format for CSS usage, so I introduced lookup tables to handle these cases. I also converted numeric values to `px`, with the exception of line heights and weights.

The expected workflow for normal changes, such as variable renaming or colour updates, is for the developer to update the Figma JSON, run the extraction script using `npm run figma-tokens:extract`, and then manually copy the generated themes into the global CSS. I avoided writing directly to the global CSS so developers could review the output before applying it.

If additional brands are added to the Figma tokens, the current solution requires adding their keys to the `brands` array in the extraction script. This could become tedious at scale. A possible improvement would be to automatically derive brand keys from first level objects following the `Mapped/{Brand}` pattern, which would allow full automation of brand handling.

There are several limitations in the current implementation. It is fragile, as circular references would cause infinite loops. Deep structural change to the JSON could also require code changes. Future improvements could include better invalid path handling, additional lookup tables, and more edge case coverage. The script also duplicates responsive values per brand, even though these values are not brand specific. This could be fixed by extracting responsive logic into a separate step and placing those variables into a shared `:root` block. 

One **MAJOR** limitation is the lack of memoization during reference resolution. If multiple tokens point to the same primitive (e.g., several brand colors referencing `Colours.red`), the script performs a full traversal for each instance. In a production environment, I would implement a cache/lookup map to store resolved values. I believe this would offer significant performance improvements.

In a production environment, a more robust custom script could adopted along with a CI process that syncs tokens between figma and the app. More of a long shot, but a script could be written to flag stale tokens used in classes. Alternatively Style Dictionary could be explored for a more scalable solution.

## Component Building

Once the themes were generated and applied, I moved on to building the components. I set up a global theme switcher and a theme wrapper component for storybook. I also installed `clsx` to simplify conditional class handling.

For component architecture, I aimed to keep component props immutable. I enforced this by using TypeScript’s Readonly utility type and destructuring props as `const`. This can be time consuming, but with Copilot it was mostly handled through autocomplete after defining the props interface. When styling components and following the Figma documentation, I used a mobile first approach. I relied on Tailwind custom variables to consume the theme values. (e.g: `text-[var(--custom-var)]`, `[font-size:var(--custom-var)]`)

The Button component was relatively straightforward. The main complexity was handling variants in a clean and scalable way, as well as managing icons. I used a lookup object for variants, where each variant defined its `default`, `hover`, and `disabled` classes. If an invalid variant is passed(i.e. not any of `primary`, `secondary`, or `tertiary`), it falls back to the `primary` variant. For icons, the consumer passes an icon key(e.g. `user`, `chevron-left`) which is resolved internally. This ensures control over which icons are used across the system. Icons are hidden from screen readers. Props are forwarded so the component behaves like a native button.

The `Card` component consist of the `Button` component so it made sense to do that next. Based on the Figma designs, which showed different call to action text depending on context(Card vs Login drawer showcase), I exposed the text values and callback through the component's API. To achieve pixel accuracy, I copied the SVG directly from Figma and wrapped it in a dedicated component. I made the SVG's colours dynamic using Tailwind classes combined with CSS variables so it responds to theme changes. I attempted to align the SVG border radius with theme values, but could not achieve pixel perfect results. I chose to keep the original SVG with dynamic colours. Further SVG manipulation could improve this in the future.

The `Input` component required careful attention to accessibility and state handling. All native input props are forwarded so the full input API is available. I limited the type prop to `text` and `email` at the TypeScript level, as these were the most relevant. This restriction can be bypassed, so proper runtime handling of unsupported types would be a future improvement.

The `IconButton` component was not explicitly part of the Figma designs, but I introduced it to avoid repeatedly wrapping icons in buttons. This component enforces the use of an aria-label so the button always has an accessible name. It only requires an icon key from the consumer.

The `Dashboard` component was the most complex. I ensured it was fully accessible, including keyboard navigation with arrow keys. I struggled to match the scrollbar thumb styling to the Figma design and consulted AI for guidance on this specific part.

The `LoginDrawer` presented architectural challenges. I debated between a controlled approach using `isOpen` and `onClose` props and an imperative global system where the drawer always lives at the top of the document. I chose the controlled approach as it was easier to reason about within the given time. Once implemented, I added focus trapping while the drawer is open to support proper forward and backward keyboard navigation. In a production environment, I'll likely explore an imperative way of handling this (Most likely with a hook, portals, and probably a context provider) so modals/dialogs/drawers can be opened anywhere and will always on top of the page (Structurally and visually).
